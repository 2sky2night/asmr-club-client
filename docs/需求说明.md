## 应用版本规范

本项目采用语义化版本控制（Semantic Versioning），格式为 **主版本号.次版本号.修订号**（Major.Minor.Patch）。

### 版本递增规则
- **主版本号 (Major)**：当发生不兼容的 API 修改、重大功能重构或界面整体改版时递增。递增时，次版本号和修订号归零。
- **次版本号 (Minor)**：当新增向下兼容的功能或进行日常迭代优化时递增。递增时，修订号归零。
- **修订号 (Patch)**：当仅修复向下兼容的 Bug 或进行微小调整时递增。

### 示例
- `1.0.0`：初始正式发布版本。
- `1.1.0`：新增音频定时关闭功能。
- `1.1.1`：修复某机型无法播放 `.m4s` 格式的问题。
- `2.0.0`：播放器 UI 全面重构或核心逻辑重大变更。

### 注意事项
1. 版本号中的数字建议控制在 3 位以内。
2. 发布版本前需同步更新 `pubspec.yaml` 中的 `version` 字段。
3. 每个正式版本应配套生成 `CHANGELOG.md` 更新日志。

## 【v0.0.1】需求介绍

这是一个音乐播放软件，能够指定目录进行媒体扫描，扫描完成汇集成播放列表，在播放列表中点击列表项即可进行音乐播放。

### 要求

设计：

- 简单清爽
- 设计因基于行业常见的手机尺寸

编码：

- 风格清晰，需要有注释

| 字段           | 写入内容                                      |
| -------------- | --------------------------------------------- |

| 音乐的封面图   | 解析出的 cover                                |
| 音乐作者       | 解析出的 owner_name                           |

### 图片资源缓存

现状：用户体感上，在会话内（应用生命周期内）图片资源会因为在视口消失再出现重新加载资源，导致播放列表的封面图反复加载，造成不必要的开销。

目标：

- 能够像浏览器那样，相同的资源存在缓存逻辑，重复的资源地址不要再重新加载直接读取内存或者磁盘数据。
- 需要在设置页面提供一个开关，告知当前缓存数据以及清空缓存的按钮



因为我不了解flutter技术生态，你作为技术需要进行调研，若有任何问题或者更好的方案可以找我确认。

### 播放列表体验优化

#### 播放列表新增搜索功能

在播放列表顶部增加一个输入框

- 对播放列表进行模糊搜索
- 搜索历史
  - 在输入框聚焦时候，展示搜索历史（建议走sqlite持久化存储）
  - 搜索历史是一个scroll-view超过一定高度支持滑动查看
    - 上限30条，多余截断不展示
    - 点击搜索历史项，可以直接进行快捷搜索

#### 滑动到顶部

滑动到顶部以悬浮球的形式展示。

展示逻辑：

- 出现：当用户滑动高度超过1屏时，可以在页面右下角展示一个悬浮球
- 消失：当用户滑动高度小于1屏时，悬浮球消失
- 悬浮球消失和出现都需要过渡动画

点击逻辑：点击即可回到顶部，需要滑动的过渡动画

### 沉浸式播放器体验优化

需求背景：用户心智。由于用户大多都用过市面上常见的播放器，习惯打开沉浸式播放器时想要通过按返回键关闭沉浸式播放器。

需求内容：打开沉浸式播放器时，通过按返回键关闭沉浸式播放器。而不是弹出退出应用的提示。
