# Flutter Code Review Skill

## 触发条件
当用户请求对 Flutter 代码进行评审，或提交代码前希望获得自动化反馈时调用此 Skill。

## 执行流程

1. **获取代码变更**：
   - 使用 `git diff` 获取工作区和暂存区的差异。
   - 重点关注 `.dart` 文件的变更。

2. **静态分析与规范性检查**：
   - **命名规范**：检查变量、函数、类名是否符合 Dart 风格指南（camelCase, PascalCase）。
   - **组件化原则**：检查 UI 逻辑是否过度耦合，建议提取可复用组件。
   - **常量提取**：识别硬编码的字符串、颜色或尺寸，建议提取为常量或主题配置。
   - **异步处理**：检查 `Future` 和 `Stream` 是否有适当的 `await` 和错误处理（`try-catch`）。

3. **安全性与性能审查**：
   - **内存泄漏**：检查监听器（如 `StreamSubscription`）是否在 `dispose` 中正确取消。
   - **隐私保护**：确保文件扫描逻辑严格校验 `rootPath`，防止跳出指定目录（参考项目 Agents.md）。
   - **构建性能**：避免在 `build` 方法中执行耗时操作或频繁创建对象。

4. **项目特定规范整合**：
   - **技术栈一致性**：确认状态管理使用 `provider`，音频播放使用 `just_audio`。
   - **路径处理**：验证是否使用了 `_normalizePath` 处理 Android 路径重复问题。

## 评审清单 (Checklist)

- [ ] 命名是否符合 Dart 官方风格？
- [ ] 是否有未处理的异常或空的 `catch` 块？
- [ ] UI 组件是否过于庞大（建议拆分）？
- [ ] 是否遵循了项目的隐私保护准则？
- [ ] 是否引入了不必要的依赖或重复代码？

## 输出格式

请按以下结构提供评审意见：

### 📄 文件名: `path/to/file.dart`

**✅ 优点**
- 描述代码中做得好的地方。

**⚠️ 潜在问题**
- 指出可能存在的 Bug、性能瓶颈或安全隐患。

**💡 改进建议**
- 提供具体的代码修改建议或重构方案。

---
*注：如果没有发现明显问题，请简要说明代码符合规范。*
