# 媒体扫描性能优化方案

## 1. 现状分析与缺陷
当前 `media_scanner.dart` 存在以下严重性能瓶颈：
- **高频跨进程通信**：每进入一个目录都需要调用一次 `listFilesNative`，每找到一个音频文件又要向上递归调用多次 `readEntryJsonNative`。在深层目录下，这种“一来一回”的通信开销是扫描慢的根本原因。
- **Isolate 通信成本**：虽然使用了 `compute` 将扫描放入后台，但 Isolate 与主线程之间的数据拷贝（尤其是大量 `Music` 对象）也会占用时间。
- **重复 I/O 操作**：Dart 层的 `_processAudioFileWithMetaStatic` 逻辑较为复杂，且在发现 MP3 时还会进行额外的文件存在性检查，导致 I/O 次数翻倍。

## 2. 优化目标
- **速度提升**：将扫描速度提升 5-10 倍，实现秒级扫描数千个文件。
- **降低耦合**：简化 Dart 层逻辑，使其仅负责发起请求和接收最终的音乐列表。
- **保持兼容性**：确保路径去重、隐私校验（不跳出根目录）和元数据提取逻辑不变。

## 3. 实施步骤

### 任务 1：重构 Android 原生层 (MainActivity.kt)
- **新增方法**：在 `MainActivity.kt` 中实现 `scanBilibiliCacheNative` 方法。
- **批量处理**：
  - 使用 Kotlin 的原生 `File` API 或 `DocumentFile` 进行深度优先遍历（DFS）。
  - 在原生层直接完成 `entry.json` 的查找与解析。
  - 将找到的音乐信息封装为 `List<Map<String, String>>`。
- **路径校验**：在原生层增加 `rootPath` 校验，确保不会扫描用户未授权的目录。

### 任务 2：更新 MethodChannel 接口
- 在 `MainActivity.kt` 的 `setMethodCallHandler` 中注册新的 `scanBilibiliCacheNative` 通道。
- 确保返回的数据结构能被 Dart 侧正确反序列化。

### 任务 3：精简 Dart 侧扫描逻辑 (media_scanner.dart)
- **移除递归逻辑**：删除 `_scanDirectoryRecursiveStatic` 和 `_processAudioFileWithMetaStatic` 等复杂的 Dart 侧递归代码。
- **单次调用**：修改 `scanBilibiliCacheStatic`，使其直接调用原生的 `scanBilibiliCacheNative`。
- **数据转换**：将原生返回的 `List<dynamic>` 快速转换为 `List<Music>` 对象。

### 任务 4：数据库写入优化
- **批量插入**：确保 `DatabaseService` 中的 `insertMusics` 使用了事务（Transaction）处理，避免逐条插入带来的 SQLite 锁竞争。
- **去重前置**：由于原生层已经做过初步筛选，Dart 侧只需进行一次性的数据库路径比对即可。

## 4. 预期效果
- **扫描耗时**：从原来的数分钟缩短至数秒。
- **UI 响应**：由于繁重工作全部在原生后台线程执行，Flutter UI 将保持绝对流畅。
- **代码维护**：Dart 侧代码量减少约 60%，逻辑更清晰。
